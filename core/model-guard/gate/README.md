# model-fit gate — Red-Green 회귀 픽스처

`model-fit-windows.json`은 model-advisor 판정의 **동결된 정답(ground truth)**이다: 16개 세션에서
뽑은 실제 12턴 윈도우 **35개**를, 3명의 독립 라벨러가 {구현/설계/단순질문/모호}로 라벨링한 뒤
다수결(`consensus` 필드)로 확정한 것 — **14 구현 / 20 설계 / 1 모호** (라벨러 일치 Fleiss κ =
0.926, "almost perfect"). 각 윈도우는 훅이 트랜스크립트 꼬리에서 계산 가능한 `features`만 담는다
(대화 내용은 포함하지 않음).

`test/model-fit-gate.test.mjs`가 `features → verdict` 분류기를 이 정답에 대고 **세 경계**로
채점한다. 하나라도 문턱 미달이면 **RED**(변경 차단), 전부 통과하면 **GREEN**:

1. **라우팅** — 구현 recall ≥ 0.95 AND 설계 recall ≥ 0.85 (작업을 신뢰성 있게 감지).
2. **§8-d 무넛지(하중)** — 20개 설계 윈도우 중 다운그레이드 넛지를 받는 게 **≤ 1**. 게이트 코퍼스가
   증명한 것: 다운그레이드 대상("대화형") 버킷은 **20 설계 : 0 단순질문** — 강한 모델이 값하는
   경우가 ~전부고 잡을 단순질문은 ~0. 그래서 그 버킷 넛지는 ~95% 틀리고 ~0% 유용. 비구현 넛지를
   되살리는 어떤 변경도 이 경계를 **RED**로 만든다.
3. **무해(do-no-harm)** — 비구현 버킷에 다운그레이드 넛지 0건 (정답에 단순질문이 없어 정당화할 근거
   자체가 없음).

## 왜 존재하나

**단순질문 vs 설계는 영속된 트랜스크립트에서 판별 불가능**하다 — 105개 라벨러 판정 중 "단순질문"은
1번 붙었고 합의 0건. 그러니 기준은 그 경계를 쫓으면 안 된다. 판정 기준 상세와 이 결론의 근거는
[`../docs/model-fit-criteria.md`](../docs/model-fit-criteria.md) §9 참조.

## 확장/재생성

라벨된 윈도우를 `{id, model, consensus, features}` 스키마로 JSON에 추가한다. 새 기준을 채점하려면
`test/model-fit-gate.test.mjs`의 `classify(features)`를 교체하면 된다.
